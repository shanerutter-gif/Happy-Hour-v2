# spotd — Full Audit (2026-07-09)

Eight parallel audits: design, features, marketing, SEO/GSC, admin + business portal, performance, growth (re-engagement + acquisition), and Supabase. Every finding below was either verified live (curl / headless Chromium / live SQL against the production DB) or backed by cited research. Where a claim mattered most (security, SEO 404s), it was independently re-verified.

## State of the union (live numbers)

139 users · 27 new signups in the last 30 days · ~16 MAU · 14 users have ever checked in · 1,922 active venues across 7 cities (data quality is genuinely strong: 97%+ photo'd, 95%+ with deals). The product is feature-complete far beyond its user base; the constraint is not features — it's distribution, activation, and two production-breaking regressions found below.

## 🔴 DO THIS WEEK — verified critical issues

1. **SECURITY: the Supabase `service_role` key is publicly readable** at `www.spotd.biz/admin.html` (line ~4079) and `/admin/board.html:200` — verified live, unauthenticated. It bypasses ALL row-level security, expires in 2088, and also authorizes `POST /api/run-migration` (arbitrary SQL execution). Additionally, admin login is theatre: any signed-in user reaches the full admin (no allow-list check in `handleLogin`). **Action: rotate the key in Supabase → update Vercel env vars → remove it from both HTML files → route admin writes through JWT-checked endpoints (the `api/admin-enrich-venues.js` pattern).**
2. **SECURITY: anyone can take ownership of any venue.** `approve_venue_claim()` is SECURITY DEFINER, executable by the anon role, with no admin check inside (verified by reading the live function body). Combined with the open claim-insert policy: insert claim → self-approve → become `owner_verified`. **Action: add `is_giveaway_admin()` guard inside + revoke EXECUTE from anon/authenticated.**
3. **SECURITY: any logged-in user can rewrite every venue.** `venues` UPDATE policy is `USING (true)`; same for events insert/update/delete. One malicious signup could deface all 1,922 listings. **Action: owner-scoped policies.**
4. **SEO: ~900 of 1,913 venue pages 404 in production** — every unpaginated Supabase fetch silently hit PostgREST's 1,000-row cap when the 7-city launch pushed active venues past 1,000. The sitemap emits exactly 1,000 URLs (913 missing) including URLs that 404; `/spots` omits San Diego and Orange County entirely. Verified live. **Action: paginate/target the fetches in `api/spots.js`, `api/sitemap-venues.js`, `api/sitemap-cities.js`, `api/spots-directory.js` (details in SEO section).**
5. **RLS is disabled on 8 tables** including `conversations` (which already has policies written — just never enabled) and `newsletter_subscribers`. **Action: one-line `ENABLE ROW LEVEL SECURITY` per table.**
6. **Marketing plumbing: `api/loops-event.js` sends event properties at the top level instead of `eventProperties`** — every property-carrying client event arrives in Loops blank. One-line fix that unblocks all behavioral email.

## The growth thesis (synthesis across all 8 audits)

spotd's biggest lever is NOT new features. Three compounding moves:

1. **Repair the pipes (week 1-2):** the six critical items above + immutable cache headers + Smart App Banner meta on all SEO pages. The SEO fix alone roughly doubles indexable inventory; the security fixes protect everything else.
2. **Close the activation loop (weeks 2-6):** personalize the first feed from onboarding answers (vibes/neighborhood are currently collected then thrown away), add a pre-check-in activation ladder (save 3 spots → follow a hood → night plan), configure the empty `push_automations` engine (HH-starting-at-your-saved-spot is the killer trigger), make the referral two-sided, and put `?ref=` on every share.
3. **Concentrate, then template (weeks 6-13):** don't market 7 cities at once with ~16 MAU. Win one beachhead neighborhood (supply: claim 20 venues via CRM sprints; demand: 3 micro-creators + Reddit + the existing blog engine), measure with the funnel dashboard, then replicate.

Everything below maps to existing infrastructure — no new vendors, no frameworks, no parallel systems.

# 1. Design audit

Method: full read of design tokens + key CSS/JS surfaces; live render of www.spotd.biz in headless Chromium (393×852 mobile, incl. dark theme); WCAG contrast math on actual token values; research-backed.

## Top 10 issues

1. **First-open gate pile-up.** A fresh visitor sees the 21+ age sheet + consent banner stacked on it + the app-download banner (`index.html:742`, z-index 9999, re-shown every visit) floating over ALL 8 onboarding screens — including squatting under "Skip for now" on the auth wall. Users who reach value within 60s are 3-5× likelier to retain (VWO, Userpilot). Fix: hide the app banner while onboarding is visible; 7-day localStorage dismissal instead of per-session; merge the age gate into onboarding screen 0 with a one-line 21+ disclosure.
2. **`--muted` fails WCAG AA everywhere — 2.72:1 on cream** (`style.css:43`, needs 4.5:1). It styles nearly every secondary label, much of it 10-12px. Coral-as-text also fails (2.82:1 both directions). Fix: raise `--muted` alpha 0.45 → 0.62 (≈4.6:1); use `--teal-dk` for coral text on light; keep coral for large/bold type and fills.
3. **The deal — the entire product — is the least legible thing on the standard card.** `.card-std-deal` is 11.5px at opacity .7 (`style.css:3103`) while the venue name gets 14px/700. The card's job is the micro-decision "is this deal worth it?" Fix: invert the hierarchy — deal to 13px/600 full opacity, demote cuisine/neighborhood.
4. **~370px of header chrome before the first venue, with two competing orange gradient banners** (giveaway banner + "Plan my night out" CTA within 200px). When everything shouts, nothing does. Fix: show one (rotate, or collapse giveaway to a gift icon after first view); consider the planner CTA as an inline feed card after position ~6.
5. **Keyboard/semantic accessibility: every venue card is a `<div onclick>`** — no role/tabindex/focus style (`app.js:3526`). With desktop two-pane now real, the core browse loop is pointer-only and VoiceOver-hostile. Fix: `role="button" tabindex="0"` + keydown on the delegated `#cardsGrid` listener + `:focus-visible` ring.
6. **White venue names wash out on light photos** in hero/compact cards (verified live). Fix: deepen the scrim gradient (start ~20%, end 0.85 alpha) or localized text-block scrim.
7. **Onboarding fabricates data** — random "47 people out right now" ±2 every 4.5s, random "Most popular tonight" badge, random "Ends in 2h 34m" timers (`onboarding.js:525,570,380`). You HAVE real live-deal counts (`hhActive`) and real check-ins; fake numbers are a trust liability in an app whose happening-now filter deliberately refuses to fake liveness. Fix: hydrate from a cheap real query; drop or compute the timer from a real venue window.
8. **Night planner buries its controls under an alphabetical wall of 40+ neighborhood chips** — 1-venue exurbs rank above Gaslamp (45). Fix: sort by count desc, top 10-12 + "Show all."
9. **Two design languages coexist: claymorphism tokens vs liquid glass.** Token block is clay; everything since June 2026 is glass. Plus 25+ distinct border-radius values and 46 `!important`s from append-at-end overrides. The app reads as three eras. Fix: pick glass (where all recent investment went), one conversion pass on primary surfaces, collapse radii to a 4-step scale (10/14/20/999). Highest-leverage polish investment available.
10. **Dark theme parity holes from hardcoded light colors** — e.g. the results count ("522 OF 522 VENUES") is invisible in dark because `.results-bar` hardcodes `rgba(42,31,20,0.35)` (`style.css:303`). Fix: sweep for hardcoded `rgba(42,31,20`/`#2A1F14`/bare `#fff` text outside tokens.

## Quick wins (<1hr each)

1. `style.css:303` — `.results-bar` color → `var(--muted)` (fixes invisible-in-dark results count).
2. Raise `--muted` alpha 0.45 → 0.62 in both themes — one token change moves hundreds of labels to AA-passing.
3. App banner: skip while onboarding overlay visible + 7-day localStorage cooldown.

## Genuinely excellent — don't touch

- Empty states are best-in-class (city-aware "Be the first in San Diego" + CTA, escape hatches, illustrated SVGs).
- CSS/JS performance discipline (content-visibility, capped staggers, the no-backdrop-filter-per-card rule, reduced-motion coverage).
- The onboarding SEQUENCE (city → live preview → vibes → attribution → social → auth) is textbook micro-commitment design; only its data honesty and gate stacking need work.
- 16px inputs (no iOS zoom), 44px tap token, and the coral→amber gradient as the consistent primary-action signature — that gradient is the brand.

# 2. Feature audit

## Inventory summary

spotd is remarkably complete for a solo-founder app. **Consumer:** 7-city discovery feed (tiered cards, two-pane desktop), concept-aware fuzzy search (`CONCEPT_MAP`), "Happening Now" live-HH filter with countdowns (`hhActive`), Night Planner bar-crawl builder, map + clusters, favorites/lists, check-ins with streaks, photo/text/story social feed with likes/comments/tags/DMs/immersive viewer, follows + leaderboards, referral codes + weekly giveaway, iOS APNs push (instant/scheduled/automations), notifications bell with badge clearing. **B2B:** venue claims, Stripe pro subscriptions, business portal. **Growth:** Loops email lifecycle, 30+ SEO/blog pages, in-house analytics, 24-tab admin portal. Confirmed against ~380 functions in `js/app.js`.

## Top 10 recommendations (impact ÷ effort, descending)

1. **One-tap "Claim this deal" redemption flow.** Competitor gap: HappyHopper ships "Exclusive Deals — claim special offers"; Crawler NYC lets users "claim offers directly in the app." spotd already stores `venues.promo_code`/`promo_description` and has `copyPromo` — but no claim event, so venues get zero proof anyone used it. Add a "Claim" button on the modal that logs a `deal_claimed` event through the existing `track()` → `/api/track-event` pipe and shows the code full-screen for the bartender. Simultaneously the consumer hook and the B2B attribution story (#2).
2. **Venue-owner analytics in the business portal.** Yelp's Upgrade Package monetizes exactly this. spotd already captures everything needed — `venue_modal_opened` (with name+city) in `analytics_events`, plus `check_ins`, `favorites`, `reviews`. A "Your month: 340 views, 22 saves, 9 check-ins, 4 deal claims" panel in `business-portal.html` (one new `ae_venue_stats` SECURITY DEFINER RPC gated on `venues.owner_id`) is the single strongest justification for the Stripe pro tier — near-zero new infrastructure.
3. **"Happy hour starting near you" push automation.** Personalized time+location pushes lift reaction rates up to 400% (CleverTap/MoEngage). spotd has all the parts: `hhActive`/`parseHHWindow` parsing, `profiles.city_slug`, `push_automations` + 15-min `push-runner` cron. Add one `trigger_type: 'hh_starting_soon'` that pushes a user's *favorited* venue when its window opens. Far higher signal than a daily broadcast.
4. **Venue/local badges on top of check-ins.** Untappd's core retention engine is badges — with *local* badges functioning like a rewards club (Untappd Lounge, Braze case study). spotd already has streaks + `checkin_streak` milestones, `showBadgeInfo` scaffolding — and a `user_badges` table with 28 rows already exists in the DB. Compute badges from `check_ins` history (Neighborhood Regular: 5 check-ins in one hood; City Explorer: 10 distinct venues; Trivia Night Hero: 3 trivia venues — amenity booleans make this trivial) and render on profiles. Later, pro venues can sponsor a badge reward = another B2B lever.
5. **Weekly leaderboard cycle + Monday push.** Apps with social comparison see up to 60% higher session stickiness; streaks + weekly leaderboards drive Week-2/4 activity (StriveCloud, StudioKrew). spotd's leaderboard is all-time — new users can never catch up, killing the motivator. Scope it to `current_week_start_pt()` (already exists for the giveaway) and announce weekly winners via a recurring `push_campaigns` row.
6. **Friends' rating on venue cards/modal.** Beli's growth engine (75M reviews, 80% of users under 35) centers on the "friend rating." spotd has `reviews` + `user_follows`; a "★ 4.5 from 3 friends" line beside `google_rating` in `renderModal` is one join and makes following actually change discovery — the missing payoff of the whole social graph.
7. **"Report outdated deal" one-tap flag.** Data freshness is existential for HH apps (AppyHour's entire pitch is live specials; spotd's own fabricated-9am-happy-hours incident proves the risk). Add a "Deal wrong?" link in the modal writing to the existing `feedback` table with `venue_id`; surface in the admin Review Queue. Cheap crowdsourced QA that compounds with scale.
8. **Persist + share night plans via DMs.** HappyHopper's headline feature is "save spots to Tonight's Plan and let friends help decide." spotd's Night Planner output currently evaporates; `npSharePlan` shares plain text. Send the plan into the existing DM system (extend the venue-card message type to a multi-stop plan card). Partiful's 400% YoY growth shows social-first going-out coordination is *the* Gen-Z behavior (CNBC) — spotd already owns venues + chat, the two halves of that loop.
9. **Guarantee Day-1 feed value for new users.** Untappd retained new users by letting them browse strangers' check-ins for Day-1 value (Braze). The 5 newly launched cities have near-empty social feeds. Cheap fix: schedule 2-3 editorial posts per city per week (the `is_official` editorial account + Demo Data patterns exist) and pin them, so no city ever opens to a blank Share tab.
10. **Surface event "going" counts + digest inclusion.** spotd has an `events` table (310 of SD's events sit inactive) and a `going_tonight_threshold` push automation, but events render only as chips. Reactivate curated events, show "🔥 N going" on event chips, and add "this week's trivia nights" to the existing `weekly-digest` cron.

## Simplify or kill

1. **Consolidate three parallel "save" mechanisms.** Favorites, custom Lists, and Saved Posts are three separate save systems — the parallel-system pattern the founder dislikes. Kill custom lists (usage: 1 list, 3 items in the DB) or fold into favorites; keep Favorites + Saved Posts.
2. **Kill Stories (24h).** Instagram-grade machinery that needs a critical mass of daily posters spotd doesn't have — empty story strips signal a dead app. Photo check-ins already deliver the "I'm out tonight" moment. Removing stories also shrinks the crash-prone media-memory surface documented repeatedly in CLAUDE.md.
3. **Sweep confirmed-dead code paths.** `goToMap`, `openAddSpotForm` + the `_addSpot*`/video-compression chain, the standalone `openTagFriends`/`skipToTagFriends` flow, the `compact` feed renderer, `renderSuggestions`/`SUGGESTIONS`, `saveEditorialPost`, the inert VAPID env var. One cleanup PR with `node --check` + smoke test.

## Top 10 fixes (prioritized)

**P0-1. The PostgREST 1,000-row cap has broken venue SEO at 7-city scale.** One root cause, three failures:
- `api/spots.js:42-48` loads only the first 1,000 of 1,922 active venues and matches the slug in-memory (`:432`) → **~900 venue pages 404 in production right now.**
- `api/sitemap-venues.js:27-30` emits 1,000 of 1,913 URLs; its filter (photo'd) differs from spots.js's (all active) so **the sitemap contains URLs that 404** — the worst trust signal you can send Google. This silently regressed the whole "522 venues stuck unindexed" recovery.
- `api/spots-directory.js:37-42` — /spots omits SD + OC entirely.
Fix: in spots.js replace fetch-all with a targeted lookup (best: persisted `slug` column + index, `?slug=eq.<slug>&limit=1`; interim: Range-header pagination). Paginate both sitemap generators and the directory. This one fix roughly doubles indexable inventory and removes live 404s. (Onely on "Discovered – currently not indexed"; Google crawl-budget docs.)

**P0-2. Slug collisions: 44 venues share a URL** (1,913 venues → 1,869 distinct slugs; e.g. `broken-shaker`, `the-abbey` in multiple cities). `slugify(name)` ignores city; `.find()` returns whichever loads first — the other venue is unreachable and the sitemap emits duplicate locs. Fix: city-qualify colliding slugs (`broken-shaker-miami`); dedupe the sitemap.

**P1-3. Split the venues sitemap by city + honest lastmod.** One sitemap per city (`/api/sitemap-venues?city=san-diego`) gives per-market indexation-rate visibility in GSC (SeekLab programmatic-SEO guidance). Add lastmod to sitemap-cities (max venues.updated_at per city); stop defaulting missing updated_at to *today* in sitemap-venues.js:35 — fake-fresh lastmod erodes crawl trust.

**P1-4. Add unique editorial content to venue pages (thin-content insurance).** Google's scaled-content-abuse enforcement (Feb + June 2025 spam updates) is outcome-based; pages survive when each adds original data/context (Breakline). Unrendered unique signals already exist: `description` (68% populated), check-in counts, promo codes. Render description, add a 2-3 sentence "Happy hour at a glance" answer-box summary at top (the format AI Overviews/LLMs lift), and cross-link the venue's NEIGHBORHOOD hub in the footer (currently only city hub — hood pages have weak inbound depth).

Verify per-city that day pages aren't near-duplicates where explicit days[] data is sparse.

**P2-7. Non-blocking fonts + preconnects on the 4 SSR renderers** (the blog checklist's media=print trick was never applied there; add preconnect for fonts.gstatic.com + the image hosts).

**P2-8. E-E-A-T/review signals.** AggregateRating uses only Spotd reviews (mostly zero). Surface google_rating/yelp_rating visibly ("4.5★ on Google" — not in schema), prioritize seeding first-party reviews in launch cities.

# 5. Admin portal + business portal audit

## Architecture verdict

The serve-from-GitHub-main pattern (`api/admin-page.js`) is a clever hack that has calcified into a liability:
- `admin.html` (385KB) is fetched from the GitHub Contents API on **every** admin page load, `cache: 'no-store'`. Every open is a cold cross-service round-trip; GitHub rate limits (unauthenticated Contents API = 60 req/hr/IP) or downtime = admin 502, no fallback.
- Branch edits to admin.html never deploy — a documented repeated footgun (Content Calendar bug, blog-editor facade).
- The 385KB monolith + 6 injected scripts is hard to reason about; injection is string-replace on `</body>` with no versioning.

**Rewire recommendations (no framework, keep it simple):**
1. **Stop fetching from GitHub at runtime.** Serve admin.html as a normal static asset (it already deploys with the repo). Either keep `api/admin-page.js` as a thin injector reading the local file, or just add the 6 script tags to admin.html directly and delete the injector. Removes latency, the 502 surface, rate-limit risk, and the branch-edits-don't-deploy trap in one move.
2. **Split the monolith opportunistically** — keep migrating self-contained tabs (CRM, CMS, demo-data) into `admin-*.js` lazy-loaded on first nav click. No big-bang.
3. **Unify the auth helper.** The 6 extension scripts each redefine `session()`; only giveaway/attribution/activity implement refresh_token handling — claims/enrichment/push-center have NO JWT refresh (after ~1hr their calls 401 with no recovery). Extract one shared `admin-auth.js`.

## Security findings (ranked) — headline items verified live by the orchestrator

- **S1 CRITICAL (VERIFIED): service_role key embedded in publicly-fetchable pages; admin gating is client-side only.** `GET /admin.html` returns full HTML to anyone, containing the Supabase service_role JWT in cleartext (`admin.html:4079`, reused :5125, :5736; also `admin/board.html:200`). It bypasses all RLS and expires 2088. All admin data ops use that key (`svcHeaders()`), so login is theatre — `handleLogin()` calls `enterApp()` after ANY successful Supabase sign-in with no allow-list check; any of the 139 registered users can use the admin, and an unauthenticated attacker doesn't even need to log in — the key is in page source. **Fix: (a) rotate the service_role key NOW (assume compromised); (b) never ship it to the browser — route admin writes through server endpoints that validate Bearer JWT → ADMIN_EMAILS/is_giveaway_admin() (the `api/admin-enrich-venues.js` pattern); (c) add admin RLS policies so clients can use user JWTs.**
- **S2 HIGH: `api/run-migration.js` is a raw SQL executor authenticated by that same shipped key.** Anyone with the page source can POST arbitrary SQL (drop tables, exfiltrate auth.users). Gate behind admin-JWT or remove the endpoint; rotate the key.
- **S3 HIGH: `admin/board.html` embeds the same key**, reachable unauthenticated. Same fix.
- **S4 MEDIUM: no RLS backstop on admin tables** (board_cards, push_campaigns, crm_*) — "RLS on, no policies" is fine until the service key leaks (it has). Add explicit `is_giveaway_admin()` policies.
- **S5 GOOD:** business-portal.html and business-landing.html correctly use only the anon key.

## Business portal gaps (by revenue impact)

The funnel (search → claim → 24hr manual verify → edit → Stripe $49/mo) is coherent and free/Pro gating is real. But value is thin for $49/mo:
1. **Analytics is promised in the upsell but doesn't exist** (portal copy sells "analytics"; landing marks it COMING SOON). Views/saves/check-ins/reviews are already captured — an owner-facing panel is the #1 lever to justify and retain the subscription, and the #1 churn risk while missing.
2. **No review-response tool** — 2026 table stakes. `reviews` table exists; add owner replies + notification.
3. **Verify the portal's deal/HH editing writes to `deals[]`, not `hours`** — portal uses edit-hours-start/end fields; if it writes HH times into `hours` it contradicts the Happening Now parser (CLAUDE.md field convention).
4. No self-serve boost/featured purchase or promo redemption counts — natural upsells once #1 exists.
5. Manual 24hr claim verification is a founder-time bottleneck — add domain/email-match auto-verify before it becomes the scaling wall.

## Admin UX quick wins

- **Unified "Today" daily-ops card**: new venue requests, pending claims, unread feedback, signups today, failed cron/push — counts + deep links on the landing dashboard.
- Finish the responsive pass: CMS editor split panel still rough ≤680px.
- **Shared `sbWrite()` wrapper that toasts on non-2xx** — the Content Calendar and blog-editor data-loss bugs share the silent-failure root cause.
- Kill the /admin/board.html service-key exposure (currently a second, separately-vulnerable app).

**First load ≈ 570 KB wire / ~1.4 MB JS parse before any data or images.**

Data: `fetchVenues('san-diego')` = 522 rows, 580 KB raw / ~120 KB gzip on EVERY city entry (no client cache). `deals` alone is 109 KB. Venue photos are hotlinked from ~130 third-party hosts (Yelp CDN originals ~115 KB each, no WebP/resizing, zero preconnects). Headers: brotli ✓, but **`cache-control: max-age=0, must-revalidate` on ALL static assets** — despite ?v= versioning, every open revalidates ~10 files. `sw.js` is push-only (no asset caching), and SW doesn't run in the iOS WKWebView anyway — HTTP cache headers are the ONLY repeat-visit lever for the native app.

## Top 10 improvements (no build step, no new vendors)

1. **Immutable cache headers — highest ROI, one config change.** Assets are already ?v=-versioned; serve js/css/icons with `cache-control: public, max-age=31536000, immutable` via vercel.json headers (note: legacy `routes` and top-level `headers` can't coexist — put headers in route objects or migrate routes→rewrites). Eliminates ~10 revalidation RTTs per open; est. 300-800ms per warm start on mobile.
2. **Cache the venue payload client-side.** Per-city sessionStorage cache (~10-min TTL), render immediately, revalidate in background. Near-instant warm city entry; removes the largest data fetch from the critical path (`enterCity` app.js:2216, `fetchVenues` db.js:711).
3. **Re-host + resize venue photos.** 522 cards reference full-size originals across 130 hosts, rendered into ~130px thumbnails. The venue-photos bucket + enrichment writer already exist — migrate photos and use Supabase render transforms (`/render/image/...?width=400&format=webp`). ~60-85% image bytes saved + fixes hotlink rot.
4. **Preconnect the LCP image host (until #3).** First hero image has fetchpriority=high but its host (s3-media0.fl.yelpcdn.com) has no preconnect — LCP pays DNS+TLS after parse. One line, ~100-300ms LCP.
5. **Lazy-load Leaflet + markercluster on first map use.** ~52 KB wire / 181 KB parse on every load even for users who never open the map; initMap already try/catches missing Leaflet. −52 KB wire for the majority path.
6. **Replace the 522-UUID `.in()` review query with an RPC.** `loadReviewAverages` (app.js:6711) builds a ~20 KB GET URL; scales linearly and will hit URL limits around 700 venues. `review_averages(city_slug)` RPC = one small request. Same pattern for `fetchCheckInCounts` (db.js:882 — pulls every raw check-in row and counts client-side).
7. **Delay gtag until after first paint.** 161 KB loading from line 6 of head competes with CSS/LCP for bandwidth. Move injection to window.load; track() already buffers.
8. **Trim the venue payload.** Modal-only fields ride the list fetch (url 42 KB, address 21 KB, photo_urls 17 KB, phone, promo_*). Lazy-fetch a venue's detail row on openModal (1-2 KB, cacheable). ~35-40% of the wire payload.
9. **Split app.js cold-path features into on-demand scripts** (Night Planner, composer, immersive viewer, DM tab → `js/app-social.js` injected on first open). No build step needed; cuts boot parse on older iPhones.
10. **Add a caching service worker for web/PWA** — cache-first for versioned assets, network-first for `/`; version the cache name with ?v= and clean on activate. Makes the "PWA" claim real (web only; WKWebView unaffected).

## Violations of the repo's own documented perf rules

- **`.map-pin-label` (style.css:2423) has `backdrop-filter: blur(4px)` per map marker** — dozens of live blur surfaces on a zoomed map / desktop two-pane. Exact class of the documented 2026-06-13 WKWebView crash rule. The label bg is already near-opaque — remove the blur.
- **`.card-compact-checkin` (style.css:5279) `backdrop-filter: blur(6px)` per compact card** — not in the tier-1 exemption list; same rule.
- **`.card-hero`/`.card-compact` lack content-visibility** — only .card-std/.sf-* got the full treatment; extend with intrinsic-size hints.
- **`renderProfile` fallback (app.js:4336) hardcodes `fetchVenues('san-diego')`** — downloads the full 580 KB SD payload just to resolve names, and is WRONG for non-SD users post-7-city launch (correctness + perf bug). Batch-fetch id,name,neighborhood for referenced ids like fetchAllCheckIns does.
- Confirmed clean: brotli, defer order, non-blocking fonts/Leaflet CSS, preconnects, eager+fetchpriority LCP images, capped card stagger — documented lessons all still in place.

Bottom line: the render path is well-tuned; the wins now are transport-layer — cache headers, payload caching, and images dwarf everything else.
